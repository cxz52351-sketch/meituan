# 大学城美食地图 - 项目介绍文档

## 一、项目概述

### 1.1 核心定位
**用AI重新定义大学生的美食决策体验**

不是又一个"美团青春版"，而是专为大学生群体打造的**决策助手+社交平台**。我们解决的不是"找餐厅"，而是"今天吃什么"这个每天困扰2000万大学生的决策难题。

### 1.2 用户洞察
通过对高校学生的观察和访谈，我发现三个核心痛点：

**痛点1：信息过载导致决策瘫痪**
- 美团/大众点评信息太多，筛选成本高
- 缺少针对学生群体的本地化标签（如"一人食友好"、"适合约会"）
- 评论质量参差不齐，难以快速判断

**痛点2：社交需求未被满足**
- 想约饭但不知道谁有空（线下拼单需求）
- 想凑满减但找不到人（线上拼单需求）
- 缺少校内美食社交圈层

**痛点3：决策疲劳**
- 每天三餐都要做选择，心智负担重
- 选择困难症，需要"帮我决定"的工具

### 1.3 解决方案
**三层产品架构：**

```
┌─────────────────────────────────────┐
│  Layer 3: AI决策层                    │
│  - AI美食助手（对话式推荐）            │
│  - 智能随机（基于偏好的随机算法）       │
│  - 口味DNA匹配                        │
└─────────────────────────────────────┘
           ↓ 数据反哺
┌─────────────────────────────────────┐
│  Layer 2: 社交层                      │
│  - 拼单广场（线上凑单+线下约饭）        │
│  - 校友圈动态（打卡、请客、转盘）       │
│  - 好友干饭状态                        │
└─────────────────────────────────────┘
           ↓ 行为数据
┌─────────────────────────────────────┐
│  Layer 1: 信息层                      │
│  - 地图/列表浏览                       │
│  - 场景化筛选（7种场景+14种品类）       │
│  - 榜单（热门/新店/高性价比等）         │
└─────────────────────────────────────┘
```

## 二、核心功能设计

### 2.1 AI决策引擎（核心竞争力）

#### 功能1：AI美食助手（对话式推荐）
**技术实现：**
- 集成DeepSeek API，采用Function Calling模式
- 4个本地工具函数：搜索餐厅、获取详情、查询营业状态、智能随机
- SSE流式响应，提升对话体验

**产品亮点：**
- 自然语言交互："我想吃辣的，预算30，一个人"
- 上下文记忆：记住用户偏好，多轮对话不重复推荐
- 主动引导：当用户说"随便"时，AI会追问场景和心情

**未来优化方向（面试时补充）：**
```
当前版本：基于规则的查询工具
V2.0计划：
1. 引入Embedding技术，实现语义搜索
   - 用户输入"想吃点暖和的" → 匹配火锅/汤面/砂锅
   - 技术方案：OpenAI text-embedding-3-small + 向量数据库

2. 协同过滤推荐算法
   - "和你口味相似的同学还喜欢..."
   - 技术方案：User-based CF + Item-based CF混合

3. 强化学习优化推荐策略
   - 根据用户点击/下单/评价反馈，动态调整推荐权重
   - 技术方案：Multi-Armed Bandit（MAB）算法平衡探索与利用
```

#### 功能2：智能随机（不是真随机）
**产品逻辑：**
```python
# 伪代码展示算法思路
def smart_random(user_history, current_context):
    # 1. 排除最近3天吃过的（避免重复）
    candidates = filter_recent(all_restaurants, user_history, days=3)

    # 2. 场景匹配（如果是晚上，降低"适合早餐"的权重）
    candidates = scene_filter(candidates, current_context.time)

    # 3. 口味DNA加权
    #    如果用户70%时间吃川菜，川菜餐厅权重×1.5
    candidates = apply_taste_weight(candidates, user.taste_dna)

    # 4. 加入探索因子（10%概率推荐从未吃过的品类）
    if random() < 0.1:
        return explore_new_category(candidates)

    # 5. 加权随机选择
    return weighted_random(candidates)
```

**为什么不是真随机？**
- 真随机可能连续推荐火锅，用户体验差
- 智能随机 = 70%偏好 + 20%新鲜感 + 10%探索

#### 功能3：口味DNA系统
**数据模型：**
```typescript
interface TasteDNA {
  // 品类偏好（基于历史订单）
  categoryPreference: {
    '川菜': 0.35,
    '日料': 0.20,
    '火锅': 0.15,
    // ...
  },

  // 价格敏感度
  priceRange: 'affordable',  // budget/affordable/moderate/premium

  // 场景偏好
  scenePreference: {
    '一人食': 0.40,
    '约会': 0.30,
    // ...
  },

  // 口味标签（通过NLP从评论中提取）
  flavorTags: ['重口味', '喜欢辣', '不吃香菜'],

  // 时间偏好
  timePattern: {
    lunch: ['快餐', '面馆'],
    dinner: ['火锅', '烧烤']
  }
}
```

**生成逻辑：**
1. 冷启动：新用户通过3-5个问题快速建立初始画像
2. 数据积累：每次点餐、评价、收藏都更新DNA
3. 动态调整：最近30天数据权重更高（口味会变化）

### 2.2 社交层设计

#### 功能4：拼单广场
**两种模式：**

**模式A：线上拼单（凑满减）**
- 场景：外卖满30减10，自己只想花15
- 流程：发起拼单 → 其他人加入 → 各付各的 → 平台自动分配优惠
- 信任机制：仅限校内认证用户，降低跑单风险

**模式B：线下约饭（社交需求）**
- 场景：想去网红店但一个人不好意思/吃不完
- 流程：发起约饭 → 展示发起人资料（院系、兴趣标签）→ 到店AA
- 社交价值：扩展校内社交圈

**商业价值（面试时补充）：**
```
1. 提升订单量：拼单降低决策门槛，促进消费
2. 商户合作：
   - 为商户提供"拼单专享价"营销工具
   - 收取商户推广费（类似美团的竞价排名）
3. 数据价值：
   - 拼单数据反映真实社交关系
   - 可用于优化推荐算法（朋友喜欢的餐厅，你可能也喜欢）
```

#### 功能5：校友圈动态
**内容类型：**
- 到店打卡：照片+评价+位置
- 请Ta吃：发起请客，朋友可领取优惠券
- 转盘结果：分享随机吃的结果，增加趣味性

**产品策略：**
- 仅限校内可见，增强归属感
- UGC激励：打卡送积分，积分兑换优惠券
- 反作弊：限制每日打卡次数，人工审核疑似刷单

### 2.3 信息层设计

#### 功能6：场景化筛选
**7种场景标签：**
- 一人食：快速、不尴尬、小份菜
- 约会：环境好、私密性、适合拍照
- 聚餐：大桌、可容纳多人、菜品丰富
- 深夜食堂：营业到凌晨、宵夜友好
- 学习续命：安静、有插座、咖啡/轻食
- 快速解决：出餐快、外带方便
- 尝鲜打卡：新店、网红店、特色菜

**为什么不用传统的"人均价格+距离"筛选？**
- 学生的决策逻辑是场景驱动，不是价格驱动
- "今天想一个人安静吃饭" vs "今天预算50元" —— 前者更符合真实需求

#### 功能7：榜单系统
**8个榜单维度：**
- 热门榜：基于浏览量+收藏量+订单量综合排序
- 新店榜：开业3个月内，按好评率排序
- 高性价比：人均价格/评分，性价比最高
- 深夜食堂：营业到23:00后的餐厅
- 约会圣地：环境评分>4.5，适合拍照
- 一人食友好：有吧台/小份菜/不尴尬
- 学霸最爱：学习场景高频打卡
- 本周上升最快：浏览量增速排序

**数据驱动（面试时补充）：**
```
当前版本：静态榜单，手动配置排序规则
V2.0计划：
1. 实时榜单：每小时更新，反映最新趋势
2. 个性化榜单：
   - "你的好友都在吃"（基于社交关系）
   - "和你口味相似的人推荐"（协同过滤）
3. A/B测试：
   - 测试不同排序算法对点击率的影响
   - 指标：CTR、转化率、用户停留时长
```

## 三、技术架构

### 3.1 技术栈选择
```
前端：React 19 + TypeScript (strict mode)
构建：Vite 5
路由：React Router DOM 7
地图：Leaflet 1.9.4
AI：DeepSeek API (OpenAI兼容格式)
样式：纯CSS，无UI框架（3300行）
```

**为什么不用Next.js/Nuxt？**
- 纯前端项目，无SSR需求
- Vite开发体验更好，HMR更快

**为什么不用UI框架（Ant Design/MUI）？**
- 移动端为主，需要高度定制化
- UI框架bundle size大，影响加载速度
- 手写CSS更灵活，符合美团品牌调性

## 四、数据驱动与增长策略

### 4.1 核心指标体系
**北极星指标：周活跃用户的人均决策次数**
- 为什么不是DAU？因为用户不需要每天打开，但每次打开要能解决问题

**分层指标：**
```
L1 - 获客指标：
- 新用户注册量
- 渠道转化率（地推、社群、口碑）

L2 - 激活指标：
- 新用户7日留存率
- 首次决策时长（从打开到选定餐厅）

L3 - 留存指标：
- 次日留存、7日留存、30日留存
- 周活跃用户数（WAU）

L4 - 变现指标：
- 订单转化率（从浏览到下单）
- 客单价
- 商户推广费收入

L5 - 推荐指标：
- NPS（净推荐值）
- 分享率（分享拼单/动态到微信）
```

### 4.2 冷启动策略
**阶段1：单校验证（1-2周）**
- 目标：验证产品核心价值，PMF（Product-Market Fit）
- 策略：
  - 地推：食堂门口扫码送奶茶
  - KOL：找校园公众号/UP主合作
  - 社群：建立"吃货群"，人工运营
- 成功标准：200个种子用户，次日留存>40%

**阶段2：校内扩散（1个月）**
- 目标：达到该校10%渗透率
- 策略：
  - 拼单裂变：邀请好友拼单，双方各得5元券
  - UGC激励：打卡送积分，积分兑换优惠
  - 线下活动：美食节、探店活动
- 成功标准：2000 DAU，周留存>60%

**阶段3：多校复制（3个月）**
- 目标：覆盖3-5所高校
- 策略：
  - 校园大使计划：每校招募5-10名大使
  - 跨校拼单：支持不同学校用户拼单（扩大网络效应）
  - 商户BD：与高频餐厅合作，提供独家优惠

### 4.3 商业模式
**收入来源：**
```
1. 商户推广费（主要收入）
   - 榜单置顶：500元/周
   - 搜索竞价：按点击付费（CPC）
   - 拼单专区：按成交付费（CPS，抽佣5-10%）

2. 会员订阅（长期收入）
   - 学生会员：9.9元/月
   - 权益：无限次AI咨询、专属优惠、优先拼单

3. 数据服务（未来收入）
   - 为商户提供选址分析、菜品优化建议
   - 基于用户画像的精准营销工具
```

**成本结构：**
```
1. AI API调用：预估0.5元/用户/月（按1000次调用计算）
2. 服务器：云服务器+CDN，约5000元/月（初期）
3. 运营成本：地推、优惠券补贴，约10万元/校（冷启动期）
```

**盈亏平衡分析：**
```
假设单校2000 DAU，月活6000：
- 收入：商户推广费 2万/月 + 会员订阅 6000×10%×9.9 ≈ 2.6万/月
- 成本：AI 3000元 + 服务器 5000元 + 运营 1万 ≈ 1.8万/月
- 利润：0.8万/月

结论：单校盈亏平衡点约1500 DAU，3-5校可实现正向现金流
```

## 五、竞品分析与差异化

### 5.1 竞品对比
| 维度 | 美团/大众点评 | 小红书 | 本产品 |
|------|--------------|--------|--------|
| 目标用户 | 全年龄段 | 18-35岁女性 | 大学生 |
| 核心场景 | 找餐厅+下单 | 种草+分享 | 决策+社交 |
| 信息密度 | 高（信息过载） | 中（图文为主） | 低（精准推荐） |
| 社交属性 | 弱 | 强（但非熟人社交） | 强（校内熟人） |
| AI能力 | 无 | 无 | 核心功能 |
| 本地化 | 弱 | 弱 | 强（校园专属） |

### 5.2 差异化优势
**1. 垂直人群的深度运营**
- 美团要服务所有人，我们只服务大学生
- 可以做极致的场景化（如"考试周续命"、"失恋了想吃甜的"）

**2. AI驱动的决策效率**
- 美团需要用户自己筛选，我们用AI帮用户决策
- 从"信息平台"到"决策助手"

**3. 校内社交���络**
- 小红书是陌生人社交，我们是熟人社交
- 拼单、约饭的信任成本更低

**4. 数据飞轮**
```
更多用户 → 更多行为数据 → AI推荐更准 → 用户体验更好 → 更多用户
    ↓
更多订单 → 商户愿意合作 → 更多独家优惠 → 用户粘性更强
```

## 六、风险与挑战

### 6.1 技术风险
**风险1：AI成本过高**
- 缓解：缓存+限流+使用更便宜的模型
- 备选方案：AI失败时降级到规则推荐

**风险2：数据冷启动**
- 缓解：新用户通过问卷快速建立画像
- 备选方案：基于热门榜单推荐

### 6.2 业务风险
**风险1：用户增长不及预期**
- 缓解：加大地推力度，优化裂变机制
- 备选方案：调整目标，先做好单校再扩张

**风险2：商户合作困难**
- 缓解：先积累用户，再谈商户合作
- 备选方案：与美团/饿了么API对接，导流分佣

**风险3：被大厂抄袭**
- 缓解：快速迭代，建立品牌心智
- 护城河：校内社交关系+用户数据积累

### 6.3 合规风险
**风险1：用户隐私保护**
- 措施：数据脱敏、最小化采集、用户可删除数据

**风险2：UGC内容审核**
- 措施：关键词过滤+人工审核+用户举报机制
